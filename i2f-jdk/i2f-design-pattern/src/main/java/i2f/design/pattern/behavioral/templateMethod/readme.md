# 模板方法模式（Template Method Pattern）

> 模板方法模式是**行为型设计模式**，定义一个操作中的算法骨架，而将某些步骤延迟到子类中实现。子类可以不改变算法结构的情况下重定义某些特定步骤。

---

## 一、核心逻辑

### 1.1 算法骨架与步骤分离
模板方法模式的核心思想是**"定义固定流程，可变步骤由子类实现"**。在抽象父类中定义一个 `final` 修饰的模板方法，该方法按照固定顺序调用多个步骤方法（抽象方法、具体方法、钩子方法），从而保证算法流程的一致性。

### 1.2 好莱坞原则（Hollywood Principle）
**"Don't call us, we'll call you"** —— 子类不主动调用父类方法，而是由父类的模板方法在适当时机回调子类的实现。这种控制反转机制确保了流程的统一性和可扩展性。

### 1.3 开闭原则（OCP）
对扩展开放，对修改关闭。当需要新增一种变体时（如新增果汁制作器），只需创建新的子类实现抽象步骤，无需修改现有代码。

### 1.4 钩子方法（Hook Method）
钩子方法是一种可选覆盖的机制，提供默认实现，子类可以选择性地覆盖以改变局部行为。这为固定流程提供了灵活的定制点。

---

## 二、核心组成

模板方法模式包含以下核心角色：

| 角色 | 说明 | 本案例对应类 |
|------|------|-------------|
| **AbstractClass（抽象类）** | 定义模板方法和基本步骤方法 | [BeverageMaker](./maker/BeverageMaker.java) |
| **Template Method（模板方法）** | 定义算法骨架，使用 `final` 防止子类修改 | `BeverageMaker.makeDrink()` |
| **Abstract Method（抽象方法）** | 子类必须实现的具体步骤 | `brew()`、`pourInCup()`、`addCondiments()` |
| **Concrete Method（具体方法）** | 所有子类共用的固定步骤 | `boilWater()` |
| **Hook Method（钩子方法）** | 可选覆盖的定制点 | `customerWantsCondiments()` |
| **ConcreteClass（具体类）** | 实现抽象步骤的具体子类 | [CoffeeMaker](./maker/impl/CoffeeMaker.java)、[TeaMaker](./maker/impl/TeaMaker.java) |

### 模式结构图

```
AbstractClass（BeverageMaker）
  └─ makeDrink(): void                   ← 模板方法（final，定义算法骨架）
     ├─ boilWater(): void                ← 具体方法（共用步骤）
     ├─ brew(): void                     ← 抽象方法（子类实现）
     ├─ pourInCup(): void                ← 抽象方法（子类实现）
     ├─ addCondiments(): void            ← 抽象方法（子类实现）
     └─ customerWantsCondiments(): boolean ← 钩子方法（可选覆盖）

ConcreteClass（具体类）
  ├─ CoffeeMaker → 实现咖啡制作的具体步骤
  └─ TeaMaker    → 实现茶制作的具体步骤（覆盖钩子方法）
```

## 三、案例设计解析

### 3.1 业务场景
以**"制作饮品"**为例——制作咖啡和制作茶的流程是固定的：
1. 烧水
2. 冲泡
3. 倒入杯子
4. 添加调料（可选）

但每一步的具体实现因饮品而异：
- **咖啡**：用热水冲泡咖啡粉 → 倒入马克杯 → 加糖和牛奶
- **茶**：用热水浸泡茶叶 → 倒入陶瓷杯 → 加柠檬片

### 3.2 模式应用流程

#### 步骤 1：定义抽象类与模板方法
[BeverageMaker](./maker/BeverageMaker.java) 作为抽象类，定义了制作饮品的固定流程：

```java
public final void makeDrink() {
    boilWater();                        // 步骤1：烧水（具体方法）
    brew();                             // 步骤2：冲泡（抽象方法）
    pourInCup();                        // 步骤3：倒杯（抽象方法）
    if (customerWantsCondiments()) {    // 步骤4：钩子方法控制
        addCondiments();                //          添加调料（抽象方法）
    }
}
```

#### 步骤 2：子类实现具体步骤
- [CoffeeMaker](./maker/impl/CoffeeMaker.java)：实现咖啡制作的具体步骤
- [TeaMaker](./maker/impl/TeaMaker.java)：实现茶制作的具体步骤，并覆盖钩子方法实现动态定制

#### 步骤 3：客户端调用
客户端通过面向抽象编程，统一调用模板方法：

```java
BeverageMaker coffeeMaker = new CoffeeMaker();
coffeeMaker.makeDrink();  // 执行咖啡制作的完整流程

BeverageMaker teaMaker = new TeaMaker(true);
teaMaker.makeDrink();     // 执行茶制作的完整流程（加柠檬）
```

### 3.3 关键设计亮点

1. **流程一致性**：`makeDrink()` 使用 `final` 修饰，确保所有饮品都遵循相同的制作流程
2. **代码复用**：`boilWater()` 作为具体方法在父类中实现，避免重复代码
3. **灵活定制**：`customerWantsCondiments()` 钩子方法允许子类在固定流程中进行局部定制
4. **扩展性**：新增饮品只需创建新子类，无需修改现有代码（开闭原则）

## 四、典型应用场景

模板方法模式适用于以下场景：

### 4.1 框架开发中的生命周期管理
- **Spring Framework**：`AbstractApplicationContext.refresh()` 定义了 Spring 容器初始化的固定骨架（12个步骤），各步骤留给子类扩展
- **Servlet**：`HttpServlet.service()` 根据请求类型分发到 `doGet()`、`doPost()` 等方法
- **JUnit**：测试框架的生命周期（`@Before` → `@Test` → `@After`）

### 4.2 数据处理流水线
- **ETL 流程**：Extract → Transform → Load，每一步的具体实现因数据源而异
- **报表生成**：查询数据 → 格式化 → 渲染输出，不同报表格式共享相同流程
- **文件解析**：读取文件 → 解析内容 → 验证数据 → 存储结果

### 4.3 算法实现框架
- **排序算法**：快速排序的分区逻辑固定，但比较策略可替换
- **搜索算法**：遍历框架固定，但剪枝策略因场景而异
- **机器学习**：训练流程固定（数据准备 → 模型训练 → 评估 → 优化），具体算法不同

### 4.4 业务审批流程
- **订单审批**：提交 → 初审 → 复审 → 终审，不同订单类型的审批规则不同
- **报销流程**：填写 → 部门审批 → 财务审批 → 打款，金额不同审批路径不同
- **请假流程**：申请 → 主管审批 → HR 备案，天数不同审批层级不同

### 4.5 游戏开发
- **游戏循环**：初始化 → 处理输入 → 更新状态 → 渲染画面，不同游戏的具体实现不同
- **角色行为**：移动 → 攻击 → 防御，不同角色的行为细节不同
- **关卡加载**：加载资源 → 初始化场景 → 启动逻辑，不同关卡的加载策略不同

### 4.6 构建器与初始化流程
- **对象初始化**：分配内存 → 设置默认值 → 执行自定义初始化 → 验证
- **连接池初始化**：创建连接 → 测试连接 → 注册到池 → 启动监控
- **插件加载**：扫描插件 → 加载配置 → 初始化实例 → 注册到容器

## 五、模式优势与注意事项

### 优势
1. **代码复用**：公共逻辑在父类中实现，消除重复代码
2. **流程控制**：模板方法保证算法骨架不被子类修改
3. **扩展友好**：符合开闭原则，新增变体只需添加子类
4. **钩子机制**：提供可选的定制点，平衡一致性与灵活性

### 注意事项
1. **避免过度设计**：如果流程只有一种实现，不需要使用模板方法
2. **抽象方法数量**：抽象方法过多会增加子类实现负担，需合理拆分
3. **钩子方法滥用**：过多的钩子方法会使流程难以理解和维护
4. **继承耦合**：子类与父类存在强耦合，需确保继承关系合理

---

## 六、与其他模式的关系

| 模式 | 区别 | 联系 |
|------|------|------|
| **策略模式** | 策略模式通过组合实现算法替换，模板方法通过继承实现 | 两者都用于算法变体，但实现机制不同 |
| **工厂方法模式** | 工厂方法创建对象，模板方法定义算法流程 | 工厂方法可视为模板方法的特例 |
| **命令模式** | 命令模式封装请求，模板方法定义流程 | 命令模式可与模板方法结合使用 |

---

> **设计原则体现**：模板方法模式完美诠释了**开闭原则（OCP）**和**好莱坞原则**，是框架开发中最常用的设计模式之一。
