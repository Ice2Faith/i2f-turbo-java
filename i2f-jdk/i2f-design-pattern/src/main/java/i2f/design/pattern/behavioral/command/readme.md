# 命令模式（Command Pattern）

> **定义：** 将一个请求封装为一个对象，从而使你可用不同的请求对客户进行参数化，对请求排队或记录请求日志，以及支持可撤销的操作。

---

## 一、核心逻辑与组成

### 1.1 核心逻辑

命令模式的核心思想是**将请求封装为对象**，实现请求发送者（Invoker）与请求接收者（Receiver）的完全解耦。其本质在于：

- **请求对象化**：每个操作（如开灯、关灯、调光）都被封装为独立的命令对象，命令对象持有接收者实例和要执行的方法。
- **接口统一化**：所有命令实现统一的 `Command` 接口（`execute()` / `undo()` / `getName()`），调用者只需通过接口触发命令，无需关心具体实现。
- **状态可逆化**：命令对象在执行前保存接收者的状态快照，使得撤销操作成为可能，这是命令模式区别于其他模式的关键特性。
- **多态调度**：调用者通过 `Command` 接口统一调度不同类型的命令（控制电灯、风扇等不同设备），体现多态的力量。

### 1.2 核心组成（5 大角色）

| 角色 | 类名 | 职责 |
|------|------|------|
| **抽象命令（Command）** | `Command.java` | 定义命令的统一接口，声明 `execute()`、`undo()`、`getName()` 方法 |
| **具体命令（Concrete Command）** | `LightOnCommand.java`<br>`LightOffCommand.java`<br>`DimmerCommand.java`<br>`FanOnCommand.java` | 实现 `Command` 接口，持有接收者引用，封装具体操作逻辑，保存状态用于撤销 |
| **接收者（Receiver）** | `Light.java`<br>`Fan.java` | 真正执行命令的对象，包含与业务逻辑相关的具体操作（如 `turnOn()`、`setBrightness()`） |
| **调用者（Invoker）** | `RemoteControl.java` | 持有命令对象引用，通过 `execute()` 触发请求，通过 `undo()` 撤销命令，维护命令历史 |
| **客户端（Client）** | `Test.java` | 创建接收者、命令对象，配置调用者的按钮绑定，发起命令执行 |

## 二、案例设计解析

### 2.1 业务场景

本包以**"智能家居遥控器"**为演示场景：

- **遥控器（Invoker）**：有 7 个按钮槽位，每个按钮可绑定不同的命令。
- **家电设备（Receiver）**：客厅电灯、卧室电灯、书房电风扇，支持开关、调节亮度/档位。
- **命令对象（Command）**：开灯、关灯、调光、开风扇等操作被封装为独立命令，支持执行和撤销。

### 2.2 命令执行流程

以"按下按钮 0（客厅开灯）"为例：

```
1. 用户调用 remote.pressButton(0)
   ↓
2. 遥控器找到 slots[0] 绑定的 LightOnCommand 对象
   ↓
3. 遥控器调用 command.execute()
   ↓
4. LightOnCommand.execute() 执行：
   - 保存电灯当前状态（brightness=0, isOn=false）
   - 调用 light.turnOn()（brightness=50, isOn=true）
   ↓
5. 遥控器将命令加入 history 列表（用于后续撤销）
```

### 2.3 命令撤销流程

以"按下撤销按钮"为例：

```
1. 用户调用 remote.pressUndo()
   ↓
2. 遥控器从 history 取出最后一个命令（如 LightOnCommand）
   ↓
3. 遥控器调用 command.undo()
   ↓
4. LightOnCommand.undo() 执行：
   - 判断执行前状态（isOn=false）
   - 调用 light.turnOff() 恢复到关闭状态
   ↓
5. 命令从 history 移除
```

### 2.4 设计亮点

#### （1）完全解耦

- **遥控器**只与 `Command` 接口交互，不知道 `Light`、`Fan` 的存在。
- **新增家电设备**（如 TV、AirConditioner）无需修改 `RemoteControl` 代码。
- **新增操作**（如 TV 换台、空调调温）只需实现新的 `Command`，符合开闭原则。

#### （2）状态快照与撤销

每个命令在执行前保存接收者的状态：

```java
// LightOnCommand.execute()
this.previousBrightness = light.getBrightness();  // 保存执行前亮度
this.previousIsOn = light.isOn();                 // 保存执行前开关状态
light.turnOn();                                   // 执行开灯
```

撤销时恢复状态：

```java
// LightOnCommand.undo()
if (previousIsOn) {
    light.setBrightness(previousBrightness);      // 恢复之前亮度
} else {
    light.turnOff();                              // 恢复关闭状态
}
```

#### （3）命令历史管理

`RemoteControl` 维护 `List<Command> history`，支持：

- **顺序执行**：`pressButton()` 将命令追加到历史。
- **逆序撤销**：`pressUndo()` 从历史末尾取出命令并执行 `undo()`。
- **历史清空**：`clearHistory()` 重置命令记录。

#### （4）多态调度

遥控器通过统一的 `Command` 接口调度不同类型的命令：

```java
remote.setCommand(0, new LightOnCommand(livingRoomLight));  // 控制电灯
remote.setCommand(3, new FanOnCommand(studyFan));           // 控制风扇
// 调用者代码完全一致：command.execute()
```

## 三、典型适用场景

命令模式适用于以下 7 类典型场景：

### 3.1 GUI 按钮/菜单操作

- **场景**：界面上的每个按钮、菜单项对应一个操作（保存、撤销、重做、打印）。
- **应用**：将每个操作封装为命令对象，按钮点击时执行 `command.execute()`。
- **价值**：按钮与业务逻辑解耦，支持动态绑定/解绑命令。

### 3.2 事务管理

- **场景**：数据库事务需要支持提交和回滚。
- **应用**：每个 SQL 操作封装为命令对象，`execute()` 执行 SQL，`undo()` 执行回滚。
- **价值**：支持事务组合、批量提交、失败回滚。

### 3.3 命令队列与延迟执行

- **场景**：任务调度系统需要将命令放入队列，按优先级或时间顺序执行。
- **应用**：将命令对象序列化后存入队列（如 `BlockingQueue<Command>`），工作线程取出执行。
- **价值**：解耦命令创建与执行时机，支持异步处理、削峰填谷。

### 3.4 宏命令（Macro Command）

- **场景**：一键执行多个操作（如"离家模式"：关灯、关风扇、锁门、开安防）。
- **应用**：创建 `MacroCommand` 持有 `List<Command>`，`execute()` 遍历执行所有子命令。
- **价值**：将复杂操作组合为单一命令，简化调用者逻辑。

### 3.5 操作日志与系统恢复

- **场景**：文本编辑器、CAD 软件需要记录用户操作，系统崩溃后恢复。
- **应用**：将每个操作（插入字符、删除行、移动图形）封装为命令对象，序列化存储到日志文件。
- **价值**：系统重启后读取日志，重放命令恢复状态。

### 3.6 远程调用（RPC）

- **场景**：分布式系统中，客户端需要远程调用服务端的方法。
- **应用**：将方法调用封装为命令对象，序列化后通过网络传输，服务端反序列化执行。
- **价值**：解耦调用方与执行方，支持跨进程、跨机器通信。

### 3.7 游戏技能系统

- **场景**：游戏中角色有多种技能（攻击、防御、治疗），技能可升级、组合。
- **应用**：每个技能封装为命令对象，`execute()` 释放技能，`undo()` 撤销技能效果（如治疗回退）。
- **价值**：技能系统易于扩展，支持技能连招、技能冷却、技能回放。

## 四、模式优势

| 优势 | 说明 |
|------|------|
| **解耦调用者与接收者** | 调用者只需知道 `Command` 接口，不需要知道接收者是谁、执行了什么操作 |
| **支持撤销/重做** | 命令对象保存状态快照，可恢复到执行前的状态 |
| **易于扩展** | 新增命令只需实现 `Command` 接口，无需修改现有代码（开闭原则） |
| **支持命令队列** | 命令对象可放入队列延迟执行、异步执行 |
| **支持宏命令** | 可组合多个命令为一个大命令，一次触发多个操作 |
| **支持日志记录** | 命令对象可序列化，系统崩溃后可通过重放命令恢复状态 |
| **符合开闭原则** | 新增命令类型对扩展开放，对修改关闭 |

## 五、相关模式

| 模式 | 关系 | 区别 |
|------|------|------|
| **责任链模式** | 可结合 | 命令模式封装单个请求，责任链模式将请求沿链传递，可组合使用（命令对象沿责任链传递） |
| **备忘录模式** | 可结合 | 命令模式用 `undo()` 撤销，备忘录模式用状态快照恢复，可组合使用（命令对象持有备忘录） |
| **原型模式** | 可结合 | 命令对象可通过原型模式克隆，支持命令模板快速创建 |
| **策略模式** | 对比 | 策略模式封装算法族，命令模式封装请求；策略模式侧重算法替换，命令模式侧重请求解耦和撤销 |

## 六、JDK/Spring 中的典型案例

| 框架 | 案例 | 说明 |
|------|------|------|
| **JDK** | `java.lang.Runnable` | 将任务封装为命令对象，线程池执行 `run()` |
| **JDK** | `java.util.concurrent.Callable` | 支持返回值的命令对象，`Future.get()` 获取执行结果 |
| **JDK** | `javax.swing.Action` | Swing 中封装 UI 操作命令，支持启用/禁用、图标、快捷键 |
| **Spring** | `JdbcTemplate#execute(StatementCallback)` | 将 SQL 操作封装为 `StatementCallback` 命令对象 |
| **Spring** | `TaskExecutor#execute(Runnable)` | 将任务封装为命令提交到线程池 |
| **Spring Batch** | `Tasklet` | 将批处理步骤封装为可执行命令，支持重试、跳过 |
| **JUnit** | `Test` 接口 | 将测试方法封装为可执行命令，测试框架统一调度 |

---

> **总结：** 命令模式通过将请求封装为对象，实现调用者与接收者的完全解耦，支持撤销/重做、命令队列、宏命令、日志记录等高级特性。它是实现"操作对象化"的经典模式，在 GUI、事务、任务调度、远程调用等场景广泛应用。
