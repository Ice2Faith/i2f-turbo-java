# 代理模式（Proxy Pattern）

## 一、模式核心逻辑

代理模式是一种**结构型设计模式**，其核心思想是：**为其他对象提供一种代理以控制对这个对象的访问**。

### 核心逻辑要点

1. **访问控制**：代理对象作为客户端与真实对象之间的中间层，可以在访问真实对象前后添加额外的控制逻辑
2. **接口一致性**：代理与真实对象实现相同的接口，客户端无需知道访问的是代理还是真实对象
3. **职责分离**：真实对象专注核心业务逻辑，代理专注访问控制、权限检查、日志记录等横切关注点
4. **透明替换**：可以在不修改客户端代码的情况下，用代理替换真实对象

### 工作原理

```
Client → 调用接口方法 → Proxy（前置处理） → RealSubject（执行业务） → Proxy（后置处理） → 返回结果
```

代理在调用真实对象之前可以执行：权限验证、参数校验、缓存检查、延迟加载等操作；在调用之后可以执行：日志记录、性能监控、结果缓存等操作。

---

## 二、核心组成

代理模式包含三个核心角色：

### 1. Subject（抽象主题）
- **定义**：定义真实主题和代理的共同接口
- **作用**：使客户端可以通过统一接口访问真实主题或代理
- **本例实现**：[IVideoPlayer](./player/IVideoPlayer.java)
  - 定义 `play(videoId)` 和 `pause()` 方法
  - 客户端面向此接口编程，不依赖具体实现

### 2. RealSubject（真实主题）
- **定义**：定义代理所代表的真实对象，包含实际的业务逻辑
- **作用**：执行核心业务功能，不关心访问控制
- **本例实现**：[VideoPlayer](./player/impl/VideoPlayer.java)
  - 实现真正的视频播放逻辑
  - 负责返回播放结果（画质、音效等业务信息）

### 3. Proxy（代理）
- **定义**：持有真实主题的引用，在访问真实主题前后添加控制逻辑
- **作用**：控制对真实对象的访问，实现权限检查、日志记录等功能
- **本例实现**：[ProxyVideoPlayer](./ProxyVideoPlayer.java)
  - 持有 `VideoPlayer` 引用
  - `play()` 方法中执行：权限检查 → 调用真实播放器 → 记录访问日志
  - 维护访问计数器 `accessCount`
  - 实现 `checkPermission()` 方法进行VIP权限验证

## 三、案例设计说明

### 场景背景
视频播放平台需要根据用户等级控制访问权限：
- **VIP用户**：可以观看所有视频（免费+付费）
- **普通用户**：只能观看免费视频（videoId以"free-"开头）

### 设计实现

#### 1. 权限控制流程
```java
// 代理的play方法核心逻辑
public String play(String videoId) {
    accessCount++;
    
    // 1. 前置处理：权限检查
    String permissionCheck = checkPermission(videoId);
    if (!"ALLOW".equals(permissionCheck)) {
        return "拒绝访问：用户无权播放";  // 拦截请求
    }
    
    // 2. 调用真实主题
    String result = realPlayer.play(videoId);
    
    // 3. 后置处理：记录日志
    System.out.println("访问日志：累计播放 " + accessCount + " 次");
    
    return result;
}
```

#### 2. 权限检查策略
```java
private String checkPermission(String videoId) {
    // VIP用户拥有全部权限
    if ("VIP".equalsIgnoreCase(userLevel)) {
        return "ALLOW";
    }
    
    // 普通用户只能观看免费视频
    if (videoId.startsWith("free-")) {
        return "ALLOW";
    }
    
    return "DENIED: 需要VIP会员";
}
```

#### 3. 客户端使用方式
```java
// 创建真实播放器
VideoPlayer realPlayer = new VideoPlayer("IcePlayer", "2.0");

// VIP用户通过代理访问
IVideoPlayer vipProxy = new ProxyVideoPlayer(realPlayer, "VIP", "VIP代理");
vipProxy.play("vip-001-复仇者联盟");  // ✓ 允许

// 普通用户通过代理访问
IVideoPlayer normalProxy = new ProxyVideoPlayer(realPlayer, "NORMAL", "普通代理");
normalProxy.play("free-001-猫咪日常");   // ✓ 允许
normalProxy.play("vip-002-阿凡达2");     // ✗ 拒绝
```

### 设计优势

1. **开闭原则**：添加新的权限规则无需修改 `VideoPlayer`，只需修改代理
2. **单一职责**：`VideoPlayer` 专注播放业务，`ProxyVideoPlayer` 专注权限控制
3. **客户端透明**：客户端通过 `IVideoPlayer` 接口访问，无需知道是代理还是真实对象
4. **灵活扩展**：可轻松替换代理实现不同的控制策略（缓存代理、日志代理等）

## 四、代理模式的常见类型

### 1. 保护代理（Protection Proxy）
- **用途**：控制对对象的访问权限
- **本例**：通过用户等级控制视频访问权限
- **典型场景**：权限管理系统、数据访问控制

### 2. 虚代理（Virtual Proxy）
- **用途**：延迟加载重量级对象
- **示例**：图片懒加载、大对象延迟初始化
- **典型场景**：
  - 网页图片延迟加载（滚动到可视区域才加载）
  - ORM框架中的延迟加载（Hibernate懒加载关联对象）
  - 大型文档编辑器（按需加载页面内容）

### 3. 远程代理（Remote Proxy）
- **用途**：代表远程对象，隐藏远程调用细节
- **示例**：RPC调用、Web Service客户端
- **典型场景**：
  - Dubbo服务调用（本地代理代表远程服务）
  - Spring Cloud OpenFeign（@FeignClient动态代理）
  - RMI（Remote Method Invocation）
  - gRPC客户端存根（Stub）

### 4. 智能引用代理（Smart Reference Proxy）
- **用途**：访问对象时执行额外操作
- **示例**：访问计数、性能监控、自动释放
- **典型场景**：
  - 智能指针（C++ shared_ptr 引用计数）
  - 数据库连接池（连接使用计数）
  - 缓存代理（访问时检查缓存）

## 五、典型应用场景

### 1. Spring AOP（面向切面编程）
- **实现方式**：JDK动态代理 / CGLIB
- **应用场景**：
  - `@Transactional`：事务管理代理（方法前后开启/提交/回滚事务）
  - `@Cacheable`：缓存代理（方法执行前检查缓存，执行后写入缓存）
  - `@Async`：异步代理（方法调用提交到线程池异步执行）
  - `@Scheduled`：定时任务代理
  - 日志切面、权限切面、性能监控切面

### 2. MyBatis Mapper代理
- **实现方式**：JDK动态代理（`MapperProxy`）
- **工作原理**：
  - 开发者只需定义Mapper接口，无需编写实现类
  - MyBatis在运行时动态生成代理对象
  - 代理对象拦截方法调用，解析SQL并执行数据库操作
- **优势**：简化DAO层开发，SQL与代码分离

### 3. 缓存代理
- **实现方式**：静态代理或动态代理
- **工作流程**：
  ```
  请求 → 代理检查缓存 → 缓存命中直接返回
                      → 缓存未命中 → 调用真实方法 → 写入缓存 → 返回结果
  ```
- **典型场景**：
  - Redis缓存代理（Spring `@Cacheable`）
  - 本地缓存代理（Caffeine、Guava Cache）
  - CDN缓存代理

### 4. 日志代理
- **用途**：记录方法调用的入参、出参、耗时、异常
- **实现**：AOP代理或手动代理
- **典型场景**：
  - 接口调用日志（出入参记录）
  - 性能监控（方法执行耗时统计）
  - 审计日志（关键操作记录）

### 5. 防火墙代理 / 安全代理
- **用途**：过滤非法请求、防止SQL注入、XSS攻击
- **典型场景**：
  - Web应用防火墙（WAF）
  - API网关（请求过滤、限流、熔断）
  - 数据脱敏代理（敏感字段自动加密/脱敏）

### 6. 延迟加载代理（懒加载）
- **用途**：推迟重量级对象的创建，直到真正需要时
- **典型场景**：
  - Hibernate关联对象懒加载（`@OneToMany(fetch = FetchType.LAZY)`）
  - 图片/视频懒加载（前端滚动到可视区域才加载）
  - 单例模式中的延迟初始化

### 7. 同步代理
- **用途**：为线程不安全的对象添加同步控制
- **示例**：
  ```java
  List<String> syncList = Collections.synchronizedList(new ArrayList<>());
  Map<String, Object> syncMap = Collections.synchronizedMap(new HashMap<>());
  ```
- **典型场景**：多线程环境下的集合访问控制

### 8. 智能指针代理（C++）
- **用途**：自动管理内存，防止内存泄漏
- **实现**：引用计数代理（`std::shared_ptr`）
- **典型场景**：C++资源管理、避免野指针

## 六、动态代理 vs 静态代理

### 静态代理（本例）
- **特点**：代理类在编译期已确定，手动编写
- **优点**：代码清晰、易于调试、IDE支持好
- **缺点**：每个目标类都需要编写对应的代理类，代码冗余
- **适用**：代理逻辑固定、目标类数量少

### 动态代理
- **特点**：代理类在运行时动态生成
- **实现方式**：
  - **JDK动态代理**：基于接口（`java.lang.reflect.Proxy`）
  - **CGLIB**：基于继承（字节码生成）
- **优点**：一个代理类可代理多个目标类，代码复用性高
- **缺点**：调试困难、性能略低
- **适用**：AOP框架、通用拦截器

---

## 七、模式优势与局限

### 优势
1. ✅ **开闭原则**：在不修改真实对象的前提下扩展功能
2. ✅ **单一职责**：业务逻辑与横切关注点分离
3. ✅ **客户端透明**：通过统一接口访问，降低耦合
4. ✅ **灵活扩展**：可动态添加/替换代理实现不同策略
5. ✅ **保护真实对象**：避免客户端直接访问，增强安全性

### 局限
1. ⚠️ **增加复杂度**：引入额外层次，增加系统理解成本
2. ⚠️ **性能开销**：代理调用会增加一层方法调用（动态代理通过反射更明显）
3. ⚠️ **过度设计风险**：简单场景使用代理可能杀鸡用牛刀

---

## 八、与其他模式的关系

| 模式 | 区别 |
|------|------|
| **装饰器模式** | 装饰器关注**功能增强**（叠加），代理关注**访问控制** |
| **适配器模式** | 适配器改变接口，代理接口不变 |
| **外观模式** | 外观简化复杂子系统接口，代理控制单个对象访问 |

---

## 九、总结

代理模式通过引入中间层，实现了**访问控制与业务逻辑的解耦**。本例以视频播放平台为场景，展示了保护代理如何通过权限检查控制用户访问，同时保持了客户端代码的透明性。

在Java生态中，代理模式广泛应用于Spring AOP、MyBatis、缓存框架、RPC框架等核心组件，是理解现代框架设计思想的重要基础。

**核心要点**：
- 代理与真实对象实现相同接口
- 代理持有真实对象引用
- 代理在调用前后添加控制逻辑
- 客户端无需知道访问的是代理还是真实对象

> 💡 **设计原则体现**：代理模式是开闭原则（OCP）和单一职责原则（SRP）的典型实践，通过代理层将变化隔离，使系统更易维护和扩展。
