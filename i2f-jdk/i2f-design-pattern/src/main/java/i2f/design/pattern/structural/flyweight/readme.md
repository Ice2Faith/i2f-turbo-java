# 享元模式（Flyweight Pattern）

## 一、模式概述

### 核心逻辑

享元模式是一种**结构型设计模式**，其核心思想是**运用共享技术有效地支持大量细粒度的对象**。通过区分对象的**内部状态**（可共享）和**外部状态**（不可共享），将相同内部状态的对象在内存中只保留一份实例，从而大幅减少内存占用。

**核心机制：**
- **共享重复对象**：相同特征的对象在内存中只创建一次，后续请求直接返回已存在的实例
- **状态分离**：将对象状态分为内部状态（intrinsic）和外部状态（extrinsic）
  - **内部状态**：存储在享元对象内部，可共享，创建后不变（如字符本身）
  - **外部状态**：由客户端管理，随场景变化，不可共享（如位置、颜色、字体）
- **工厂管理**：通过享元工厂统一创建和管理享元对象池

### 核心组成

享元模式包含以下核心角色：

| 角色 | 说明 | 本案例对应类 |
|------|------|-------------|
| **Flyweight（抽象享元）** | 定义享元对象的公共接口，声明接受外部状态的方法 | [Char](character/Char.java) |
| **ConcreteFlyweight（具体享元）** | 实现抽象享元接口，存储内部状态 | [ConcreteChar](character/impl/ConcreteChar.java) |
| **FlyweightFactory（享元工厂）** | 创建并管理享元对象池，确保相同内部状态的对象只创建一次 | [CharacterFactory](factory/CharacterFactory.java) |
| **Client（客户端）** | 维护享元的外部状态，协调享元对象的使用 | [Test](Test.java) |
| **Context（上下文）** | 存储外部状态，传递给享元对象使用 | [TextContext](context/TextContext.java) |

## 二、案例设计说明

### 场景背景

本案例以**文字排版系统**为场景，模拟文档编辑器中大量字符的渲染过程。一篇文档可能包含成千上万个字符，如果为每个字符位置都创建独立对象，将导致严重的内存浪费。

### 设计实现

#### 1. 抽象享元角色：Char

定义字符享元的公共接口：
- `render(TextContext context)`：渲染字符，接收外部状态（位置、颜色、字体等）
- `getSymbol()`：获取字符本身（内部状态）

关键设计：享元对象本身不持有外部状态，外部状态通过方法参数传入。

#### 2. 具体享元角色：ConcreteChar

实现字符的具体逻辑：
- 内部状态：`symbol`（字符本身，如 'A'、'B'）
- `render()` 方法：结合传入的 TextContext 外部状态，输出字符的渲染信息
- `getTypeDescription()`：判断字符类型（字母/数字/标点/空白）

#### 3. 享元工厂：CharacterFactory

管理字符享元池的核心组件：
```java
private final Map<Character, Char> flyweightPool = new HashMap<>();

public Char getCharacter(char symbol) {
    if (!flyweightPool.containsKey(symbol)) {
        // 不存在：创建新享元并放入池中
        flyweightPool.put(symbol, new ConcreteChar(symbol));
    }
    // 已存在：返回共享实例
    return flyweightPool.get(symbol);
}
```

**核心逻辑：**
1. 检查享元池中是否已存在该字符
2. 存在 → 返回共享实例（节省内存）
3. 不存在 → 创建新实例并放入池中

#### 4. 上下文：TextContext

存储字符的外部状态：
- 位置坐标：`x`, `y`
- 样式信息：`color`, `fontSize`, `fontFamily`

这些状态不存储在享元对象中，而是由客户端在渲染时动态传入。

#### 5. 客户端调用：Test

演示享元模式的使用流程：

```java
// 1. 创建享元工厂
CharacterFactory factory = new CharacterFactory();

// 2. 获取字符享元（相同字符共享同一实例）
Char h1 = factory.getCharacter('H');
Char e1 = factory.getCharacter('e');
Char l1 = factory.getCharacter('l');
Char l2 = factory.getCharacter('l');  // 复用 'l'，不创建新对象

// 3. 验证共享机制
System.out.println(l1 == l2);  // true - 同一实例

// 4. 渲染时传入外部状态
h1.render(new TextContext(10, 20, "BLACK", 14, "Times New Roman"));
e1.render(new TextContext(25, 20, "BLACK", 14, "Times New Roman"));
```

### 内存优化效果

案例中演示了渲染一篇文档的场景：

| 指标 | 未使用享元 | 使用享元 |
|------|-----------|---------|
| 文档字符数 | 175 个 | 175 个 |
| 创建对象数 | 175 个 | ~47 个（不同字符种类） |
| 内存节省 | - | **约 73%** |

对于 100 万字的文档：
- **不使用享元**：100 万个对象 × 64 bytes ≈ 64 MB
- **使用享元**：最多 128 个对象（ASCII 字符集）× 64 bytes ≈ 8 KB
- **节省率：99.99%**

## 三、典型应用场景

享元模式适用于以下场景：

### 1. 文字处理与富文本编辑器
- **场景**：Word、Notion 等编辑器中，文档包含大量字符
- **应用**：相同字符共享享元实例，位置/颜色/字体作为外部状态
- **优势**：大幅减少内存占用，支持大文档流畅编辑

### 2. 图形编辑器与游戏开发
- **场景**：绘制大量重复图形元素（树木、粒子效果、UI 图标）
- **应用**：相同图形共享享元，位置/旋转/缩放作为外部状态
- **优势**：减少 GPU 内存占用，提升渲染性能

### 3. 数据库连接池与线程池
- **场景**：频繁创建销毁连接/线程导致性能损耗
- **应用**：连接/线程作为享元池中的共享对象，用完归还
- **优势**：避免频繁创建销毁，提升系统吞吐量
- **典型案例**：HikariCP、Druid 连接池，ThreadPoolExecutor

### 4. 字符串常量池
- **场景**：程序中大量字符串字面量
- **应用**：JVM 内部使用享元模式实现 String Pool
- **优势**：相同字符串共享同一实例，节省内存
- **代码示例**：`String.intern()` 方法

### 5. 包装类缓存（Integer、Long 等）
- **场景**：频繁创建小整数对象
- **应用**：JDK 对常用数值范围使用享元缓存
- **优势**：减少对象创建，提升性能
- **典型案例**：
  - `Integer.valueOf(-128 ~ 127)` 返回缓存实例
  - `Long.valueOf()`、`Character.valueOf(0~127)` 同理

### 6. 图标与资源管理
- **场景**：应用中大量重复使用的图标、图片资源
- **应用**：相同图标共享享元，使用位置/大小作为外部状态
- **优势**：避免重复加载资源，降低内存占用

### 7. 缓存系统
- **场景**：热点数据频繁访问
- **应用**：相同数据对象缓存共享，如 Spring Bean 单例池
- **优势**：减少重复查询，提升响应速度
- **典型案例**：`DefaultSingletonBeanRegistry#singletonObjects`

### 8. 网络协议报文处理
- **场景**：高并发网络服务中大量相似报文对象
- **应用**：报文模板作为享元，具体数据作为外部状态
- **优势**：减少对象创建，降低 GC 压力

## 四、使用注意事项

### 适用条件
1. **大量相似对象**：系统中存在大量相同或相似的对象
2. **状态可分离**：对象的大部分状态可以外部化
3. **内存敏感**：对象创建导致内存占用成为瓶颈
4. **对象不可变**：享元对象的内部状态创建后不应改变

### 不适用场景
1. **对象差异大**：每个对象都有独特状态，无法共享
2. **外部状态复杂**：外部状态管理成本超过共享收益
3. **线程安全要求高**：共享对象需要考虑并发访问问题
4. **对象数量少**：共享带来的收益不明显

### 实现建议
1. **线程安全**：享元工厂在多线程环境下需使用 `ConcurrentHashMap`
2. **不可变性**：确保享元对象的内部状态不可变
3. **池大小控制**：必要时限制享元池大小，避免内存泄漏
4. **外部状态管理**：客户端需妥善管理外部状态，避免状态混乱

## 五、与其他模式的区别

| 模式 | 核心关注点 | 与享元的区别 |
|------|-----------|-------------|
| **单例模式** | 全局唯一实例 | 单例只有一个实例，享元可以有多个实例但同类共享 |
| **工厂方法** | 对象创建解耦 | 工厂每次创建新对象，享元相同键值返回同一对象 |
| **原型模式** | 对象克隆 | 原型创建新副本，享元返回共享实例 |
| **对象池** | 对象复用 | 对象池对象用完归还，享元对象始终共享 |

## 六、总结

享元模式通过**共享技术**和**状态分离**，在保证功能完整的前提下，大幅减少了内存占用。它是处理**大量细粒度对象**场景的利器，广泛应用于文字处理、图形渲染、资源管理、缓存系统等领域。

**核心价值：**
- ✅ 显著降低内存占用（可达 99% 以上）
- ✅ 分离关注点：内部状态共享，外部状态灵活
- ✅ 提升系统性能：减少对象创建和 GC 压力
- ✅ 适用于资源受限环境：移动端、嵌入式系统

**设计原则体现：**
- **单一职责**：享元只管理内部状态，外部状态由上下文管理
- **开闭原则**：新增享元类型无需修改工厂逻辑
- **依赖倒置**：客户端依赖抽象享元接口，而非具体实现

> 💡 **提示**：享元模式是工具而非银弹，使用时应结合实际业务场景，避免过度设计。仅在对象数量庞大且内存成为瓶颈时才考虑使用。
