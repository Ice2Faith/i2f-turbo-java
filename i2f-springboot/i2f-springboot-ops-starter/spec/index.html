Overview概述 ：运维控制台里的 AI 心脏

i2f-springboot-ops-starter 是开箱即用的 SpringBoot 运维控制台 Starter，而其中的 openai 子系统（包 i2f.springboot.ops.openai）为整个控制台装上了一颗 AI 心脏： 引入依赖即自动注册一个完整的、OpenAI 兼容的 AI 对话工作台，前端资源全部打包在 JAR 内，无需额外部署。

与常见的"瘦客户端 + 胖服务端"聊天页面不同，本框架将 Agent 状态机搬到了浏览器一侧： 前端维护会话列表、消息历史、工具审批与功能开关，后端 OpenAiOpsController 则扮演 安全网关 + 工具执行器 + SSE 中继的三重角色——每一次"发送"，都是前端把完整消息列表经国密加密后交给后端， 后端完成提示词注入、工具契约执行，再把 LLM 的流式输出原样中继回来。

框架对大模型的本质理解（摘自前端源码注释）

大模型就像是一个 CPU，Agent 等就像是程序计数器 + 程序控制块，负责准备输入、处理输出、维护消息循环、决定如何执行下一步。

换个例子：大模型就像是一个大脑，Agent 等就像是身体血肉与四肢，负责准备所见所得，处理大脑指令并反馈，维护行为连贯性。

大模型本身不具有记忆能力，更像是一个转换器——将一堆不同角色的对话历史作为输入，推理出结果。 所谓的 Tools / Function-Calling / MCP / RAG / Skills，本质都是在维护这份对话列表。

02

Design Philosophy设计理念

整个 AI 工具链遵循 i2f-turbo-java 一以贯之的"简练、独立、单一、易用"哲学，并在 AI 领域落地为四条核心原则。

壹

平台无关化

Platform Agnostic

标准层 i2f-ai-std 不依赖任何 AI 平台 SDK（OpenAI SDK / LangChain4j 等），仅依赖项目内部基础模块。 上层可自由切换 4 种 AiModel 实现：自研零依赖 HTTP 实现、OpenAI 官方 SDK、阿里 DashScope、LangChain4j。

贰

依赖无关化

Dependency Free

协议实现层 i2f-ai-rest-openai 使用项目自有 HTTP 客户端 i2f-network， 零第三方 AI/HTTP 依赖；Ops 层则复用 Spring 的 RestTemplate 作为 SSE 传输底座。

叁

接口-实现分离

std / impl Split

i2f-ai-std 定义全部接口与标准（AiModel / ToolManager / RagWorker / McpToolProvider）， 实现散布于各 extension 模块。依赖单向流动，杜绝循环依赖。

肆

胖客户端 + HITL

Fat Client · Human In The Loop

需要用户参与（工具授权、危险操作把关）的场景选择胖客户端：状态维护在浏览器，服务端无状态、可水平扩展。 全自动化场景（Dify / ComfyUI 式）才需要胖服务器——本框架明确选择了前者。

为什么是"消息列表即一切"？

多轮对话中，模型会重新读取整个对话列表——这就是模型"记忆"的最初形式。框架源码用 [1u] → [2ac] → [3t] → [4a] 记号法描述一轮 ReAct： 1u 第一条用户消息；2ac 第二条助手消息（c = 携带 tool_calls 契约）； 3t 第三条工具消息；4a 最终归纳答复。第一轮调用仅携带 [1u]（用户触发）， 第二轮携带 [1u, 2ac, 3t]（客户端自动触发）——客户端必须维护复杂状态，这正是胖客户端存在的理由。

04

Conversation Pipeline核心对话流程

POST /ops/open-ai/stream 是整个子系统最核心的端点（OpenAiOpsController#stream，约 600 行编排逻辑）。 它返回一个 5 分钟超时的 SseEmitter，真正的编排跑在 work-stealing 线程池中。

SSE 对话时序 — 后端是"安全网关 + 工具执行器 + 流中继"三位一体
编排关键决策点

解密与线程切换 transfer.recv()

请求体 OpsSecureDto 经国密链路解密还原为 OpenAiOperateDto；随后 CompletableFuture.runAsync 切入 work-stealing 池（min(max(CPU×4+2, 16), 512) 线程），HTTP 线程立即返回 SseEmitter。

提示词注入时机判定 needInjectSystemPrompt

仅当末消息是 user/system 时注入系统提示词；若末消息是 assistant/tool（多为中间过程）则默认不注入，但保留 30% 概率随机注入，保证长周期对话中提示词引导不丢失。

工具集动态裁剪

依据 enableMemories / enableTruth / enableSkills / enableRags / enableLruTools 五开关，按绑定类（MemoryTools / TruthStoreTools / SkillsTools / RagTools / McpProviderTools）过滤工具列表——功能开关即工具可见性。

工具契约执行 HITL 审批

末消息为带 tool_calls 的 assistant 消息时，逐个契约核对前端审批决定（toolApprovalList）：拒绝则注入拒绝原因作为错误结果；允许则提交 toolPool 并行执行，CountDownLatch 汇聚后回填消息列表。

流式中继 SpringWebHttpProcessor

通过 HttpRequest.doPost(url).json() 携带 Bearer 令牌请求 LLM，逐行读取 data: 前缀的 chunk，封装为 OpsSecureReturn（按 encryptOutput 决定是否加密）后 emitter.send() 中继。

05

Re-Act LoopRe-Act 循环 ：Agent 的心脏

标准层 i2f.ai.std.agent.AiAgent（471 行）实现了完整的 Reasoning + Acting 循环。 Ops 子系统虽由前端驱动循环节奏，但后端 i2f-ai-std 提供的这套 Agent 引擎是同一套设计哲学的程序化表达， 二者共享工具解析（ToolRawHelper）与消息模型。

AiAgent Re-Act 循环 — 直至模型返回 STOP 或触发护栏
消息列表演化记号法
# 一轮完整的工具调用对话 [1u] # 用户发送消息 ↓ 注入工具声明 [1u] → LLM 推理 ↓ 返回调用契约 [1u, 2ac] # 2ac = 助手消息 + tool_calls ↓ 执行工具 · 反馈结果 [1u, 2ac, 3t] # 3t = 工具消息 ↓ LLM 继续推理（客户端自动触发） [1u, 2ac, 3t, 4a] # 4a = 最终归纳答复

两轮 LLM 调用：第一轮仅携带 [1u] 由用户触发；第二轮携带三条消息由客户端自动触发——这正是"客户端即 Agent 运行时"的含义。

AiAgentContext 护栏配置
配置项 默认 说明

历史压缩策略（compressOrDropHistoryMessage）

消息数达到阈值时，将超量历史消息取出并追加一条 "总结上述对话内容" 的用户消息，单独请求 LLM 生成摘要回填—— 用一次廉价调用换取上下文瘦身；随后按 maxKeepMessageCount 硬截断，并可选保留首条用户消息以锚定对话主题。 前端页面则以"截断会话历史线"分割线可视化这一过程。

消息是框架的第一公民。标准层定义 4 种角色消息，Ops 层通过 OpenAiMessageVo 多态载体与 OpsOpenAiConsts 扩展出 7 种回显消息——让前端不仅能看到对话，还能看到"系统为模型做了什么"。

RAG Pipeline & MemoryRAG 知识库 · 记忆系统 ：向量检索与三级记忆

RAG（检索增强生成）让 LLM 基于私有文档作答：文档入库时被切分、向量化、持久化到 SQLite； 对话时由 rag_search 工具按语义相似度召回片段，作为上下文交给模型。全链路零外部中间件——一个 SQLite 文件即是向量库。

RAG 双管线 — 启动时批量入库，对话时按需检索
组件清单
组件 实现 职责
向量模型 HttpOpenAiRagEmbeddingModel OpenAI 兼容 /embeddings 接口，baseUrl / apiKey / model 可配
向量存储 SqliteRagEmbeddingStore SQLite 单文件向量库，维度可配，零中间件依赖
调度器 RagWorker load 入库 / similar 检索的统一入口
切分器 SimpleRecursiveRagTextSplitter 最大分段长度 + 重叠率，递归切分长文
工具 RagTools.rag_search LLM 主动检索知识库，默认 topN=3，建议小于 10
读取器 ListableRagFileReader Markitdown（Office）· EasyOCR（图片 / PDF-OCR）· Pandoc（文档转换）按开关组合
文档生命周期 — 加载即归档

入库完成后，loadDocs 将整个 docs 目录移动到 rags_history/history-{yyyyMMddHHmmss} 并重建空目录——同一文档永不被重复向量化，同时保留完整历史可追溯。入库在后台线程异步执行，不阻塞应用启动。

记忆系统 Memory — 跨会话的用户级长期记忆

RAG 知识库解决的是组织级文档检索，而记忆系统（MemoryTools）解决的是用户级个性化记忆： 模型在对话中主动将用户偏好、个人信息、历史结论等保存为向量记忆，后续会话经语义检索召回—— 让 AI 真正「认识」用户。记忆以 Bucket 桶隔离：默认 public 桶全局共享， 请求经 memoryBucket 字段指定用户专属桶，检索时同时覆盖公共桶与用户桶。

MemoryTools — 记忆三件套
工具 能力 标签
memory_search 按语义相似度检索历史对话记忆（偏好 / 爱好 / 个人信息等），默认 topN=3 READONLY
memory_save 保存一条记忆内容，规则：简短精确；写入用户桶（无桶则写 public） WRITABLE
memory_delete 按记忆 ID 列表批量删除 WRITABLE

记忆存储基于 SqliteBucketRagMemoryStore（带桶分区的 SQLite 向量库）， 与 RAG 共享同一个 RagEmbeddingModel 向量化服务——由 ai.rags.memory.enable 开关独立控制装配。

Bucket 桶隔离模型
# 记忆桶的读写规则 BucketRagEmbeddingStore ├── public 桶 # 全局共享记忆（默认写入桶） └── {memoryBucket} 桶 # 用户专属记忆（请求级指定） # memory_search 检索范围 检索桶 = [public] + [memoryBucket] # 即：公共记忆 + 用户私有记忆一并召回 # memory_save 写入规则 写入桶 = memoryBucket ?? public # 有用户桶写用户桶，无则写公共桶
Memory / RAG / Truth — 三级记忆架构与适用场景

框架提供三个层次互补的「记忆」机制，分别对应会话级、用户级、系统级三种信息生命周期—— 它们共享向量检索底座，但作用域、写入方式与消费方式截然不同：

维度 Truth 事实注入 Memory 记忆系统 RAG 知识库
作用域 会话级 单次聊天会话 用户级 跨会话持久，按用户桶隔离 系统级 全局共享，所有用户可见
写入者 LLM 主动调用 store_truth LLM 主动调用 memory_save 管理员投放文档，启动时批量入库
存储方式 纯文本 truthContent，前端持有 向量化 + SQLite 桶存储，服务端持久 向量化 + SQLite 存储，服务端持久

适用场景 当前任务的关键结论 / 数字 / 规则，防遗忘防幻觉 用户偏好、个人信息、历史结论等跨会话个性化记忆 组织内部文档、规章制度、产品手册等私有知识检索

协作关系 — 三层记忆如何配合

一次典型对话中三层记忆可同时生效：模型从 RAG 检索到公司报销制度（系统级知识）， 从 Memory 召回「该用户出差偏好高铁二等座」（用户级偏好）， 再把本轮计算出的报销金额经 Truth 存储（会话级事实）——后续轮次自动注入，不再重复计算。 三者由独立开关控制，按需组合：enableRags + enableMemories + enableTruth。

MCP Dynamic ToolsMCP 动态工具 ：三步发现 + LRU 装载

工具不是越多越好——上百个工具定义全量上报会吞噬 token、分散模型注意力。MCP 动态工具方案的回答是： 只上报 3 个「元工具」，让 LLM 像查目录一样先发现、再装载、按需使用，并用 LRU 控制会话内工具总量。

三步发现 + LRU 淘汰 — 工具装载的「目录-翻页」模型
注入的元提示词（McpProviderTools.SYSTEM_PROMPT）
system prompt — 动态工具查找指引 
当现有工具无法满足任务需求时：
1. 调用 tools_provider_list 列举可用的 MCP 工具供应商
                            2. 调用 list_tools_from_providers 按供应商名查看其工具清单
                            3. 调用 load_tools_by_names 将所需工具装载到当前会话
                            装载后的工具将参与后续推理；会话内工具数量受 LRU 策略约束，
                            超容量时最久未使用的工具将被淘汰。
设计要点
常驻工具不可被淘汰

MCP 元工具自身、TmpFileTools、TruthStoreTools 等基础设施类工具被标记为常驻——LRU 淘汰只作用于动态装载的业务工具，保证「查目录的能力」永远在线。

前后端双向同步

每次装载 / 淘汰后，后端经 echo_lru_tools 回显当前存活工具名列表与容量（lruToolNames / lruToolMaxSize），前端据此渲染工具面板——用户随时可见「模型手里现在有哪些工具」。

SM Crypto Transfer安全体系 ：国密全链路传输

对话报文经 OpsSecureTransfer 封装：SM4 加密载荷、SM2 加密会话密钥、SM3 摘要防篡改、SM2 签名抗抵赖、 时间戳窗口防重放——五种机制一次封装，前端 sm-crypto 与后端 i2f-sm-crypto 对称实现。

OpsSecureDto — 安全信封字段
字段 来源 安全作用
timestamp 发送时刻毫秒戳 接收方校验 ±12 小时窗口，阻断重放攻击
nonce 随机数 参与摘要计算，增加重放成本

提示词注入优先级 — 从先到后
# 后端组装报文时的注入顺序 1. 事实注入 (Truth) → messages[0] 系统消息 2. MCP 动态工具提示词 → messages[0] 系统消息 3. 技能系统提示词 → messages[0] 系统消息 4. 角色 System Prompt → 用户手动添加的角色人设 5. 用户消息 + 附件 → 消息列表末尾（<upload_files>） 6. 系统重排 (可选) → 合并全部 system 为一条置于首位
注入时机的工程考量
为什么系统提示词要「动态拼装」？

技能声明、MCP 指引、关键事实都是随功能开关与运行状态变化的活内容：技能每 30 秒可能新增，事实每轮可能被模型改写，LRU 工具集随调用漂移。因此系统提示词不能写死，而须在每次请求时按优先级重新拼装——配合 30% 概率重注入策略，保证长对话中引导不丢失。

正交开关 — 能力即插即用

上述特性分别由 enableTruth / enableEchoRequestPayload 等请求级布尔开关控制（见 OpenAiOperateDto），前端功能面板一一对应。每个开关独立生效、互不牵连——这是「消息循环为体、能力插件为用」设计哲学的直接体现。

Extension Guide扩展指南 ：三步接入一个新工具

框架的扩展面刻意保持极小：加工具只需一个注解类，加技能只需一个 Markdown 文件，加知识只需把文档丢进目录。

示例：新增一个天气工具
WeatherTools.java — 新工具仅需如此 
@Component
@Tools(tags = {AiTags.READONLY_VALUE})
                            public class WeatherTools {

                            @Tool(description = "查询指定城市的实时天气")
                            public String query_weather(
                            @ToolParam(description = "城市名称，例如：北京") String city) {
                            return weatherService.query(city);
                            }
                            }

编写 Bean

如上：Spring 组件 + @Tools / @Tool / @ToolParam 注解。

确认开关

ai.tools.enable 默认开启，ContextAppToolManager 启动时自动扫描收录。

前端验证

开启 enableEchoRequestPayload，在回显报文中确认工具定义已注入 tools 字段。

描述即接口 — 写给 LLM 的文档

description 是 LLM 决定「要不要调、怎么调」的唯一依据。写工具描述要像写 API 文档：能力边界、参数格式、返回示例、使用时机，一样都不能少；危险操作务必打上 HUMAN 标签，把最终决定权留给人。
