# 项目开发方法

## 依赖库与使用环境

- 请先查看项目根目录的 `pom.xml` 文件确定使用的依赖库版本
- 以及现有的依赖管理，现有导入的依赖
- 针对具体模块时，需要额外查看模块自身的 `pom.xml` 依赖信息，以及其引用的其他模块的依赖配置信息
- 确定运行环境（`JAVA`版本要求）

## 依赖的添加与决策

- 如果需要使用其他的依赖库的情况下
- 需要优先查看项目中是否已经有相关功能可以直接使用
- 或者项目中已经有基本具备能力稍加变更就能够实现要求的功能
- 如果没有可以直接使用的情况下
- 需要先查看现有的依赖库与运行环境（`Java`版本等）信息
- 选择一个不与现有依赖或者环境冲突的依赖库进行引入，而且必须是广泛使用经过验证的

## 代码编写

- 首先理解需求
- 然后结合现有项目分析实现方式
- 接着结合项目依赖于现有功能考虑功能复用
- 保持代码风格
- 完成编码后进行代码检查
- 然后检查安全与性能要求
- 最后需要进行代码测试
- 持续改进代码实现

## 代码变更

- 如果需要更改函数签名，例如添加函数参数，重命名函数，变更函数参数类型等这些可能破坏性的改动时
- 正确的做法是，创建一个新的函数，而不是直接改动现有函数，如果这两个函数功能上逻辑重叠的情况下，抽取公共逻辑为独立函数

## 代码检查

- 检查代码语法，保证基础语法正确，保证使用的依赖库兼容
- 检查兼容性，保证引用的方法或者类使用正确，环境兼容，调用兼容
- 检查边界，对于常见的参数边界，参数范围需要进行严格的检验，不要相信函数入参的正确性
- 检查安全性，考虑不合法的参数输入，规避不合法输入带来的安全问题
- 检查性能，考虑实现的逻辑性能，在实现基础逻辑的情况下，应该尽可能提升运行性能
- 检查空对象，根据被调用链路，判断对象是否可能为空，避免出现空指针异常，应该提前检查判断值的合法性

## 代码风格

- 复用现有，尽可能复用现有的代码功能，而不是创建一类新的或者类似的功能
- 代码命名上，按照项目命名风格，保持在项目内同一个命名尽量少的具有二义性，即使现有项目类的命名可能不太准确
- 在进行MVC（`controller/service/dao`）这种三层架构的WEB接口开发时，按照模块划分包结构
    - 例如 user 用户模块，
    - 包结构为
        - `modules/user/controller/UserController`
        - `modules/user/service/UserService`
        - `modules/user/service/UserServiceImpl`
        - `modules/user/mapper/UserMapper`
        - `modules/user/data/entity/User`
        - `modules/user/data/dto/UserDto`
- 项目内只要没有特殊要求，一律使用 `UTF-8` 编码
- 尽量使用"卫语句"提前检查参数合法性，不合法的提前返回或者抛出异常，而不是进行深层次的`if`嵌套
    - 循环中也是尽量使用"卫语句"进行判断，提前判断，进行`continue`或者`break`，而不是`if`嵌套
- 每个函数体的语句行数不要超过100行，循环体内和if语句体内同样如此，应该拆分提取为函数调用，保证单个函数内逻辑清晰简洁
- SQL编写应该尽量使用`SQL99`标准，尽量避免使用特定厂商提供的方言（Oracle，MySQL等），除非必须使用，这应该根据项目依赖引入的驱动决定使用哪种方言
- 每个类内部的功能相似或者相关度高的函数实现/变量定义应该就近定义，而不是相隔胜远
- 遵从大厂编码规范(`Alibaba Java 开发规范`、`Google Java 开发规范`等)
- 日志输出，日志默认使用Slf4j，使用logback实现，日志需要能够表现出在代码哪里因为什么导致了什么现象，避免频繁的日志，仅在关键位置进行日志输出即可
    - 在集成了`lombok`的情况下，使用`@Slf4j`注解注入`log`对象，而不是使用`LoggerFactory`获取`Logger`
    - 日志禁止输出敏感信息，例如秘钥，密码等
    - 一律禁止使用 `System.out.print*` 或 `System.err.print*` 方式输出内容
- 重试策略，当需要进行重试逻辑编写时，可以使用`spring-retry`
    - 如果不是`spring`环境，需要注意重试的指数退避间歇与最大重试次数和单次最大等待时间控制
- 数据库批量插入，最好使用`batch`模式进行批量插入，而不是循环插入每一条数据，提升性能
    - 同时，限制每个批次插入的行数，避免某些数据库驱动一次性无法处理大量行数的情况
    - 一般建议500行一个批次进行插入
- 微服务与分布式，优先考虑使用`spring-cloud+spring-cloud-alibaba`体系实现，其次考虑使用`dubbo`实现
    - 优先考虑`spring-gateway,nacos,sentinel,openfeign`
    - 针对`springboot3`及以上版本的项目，优先使用 `httpexchange` 替换 `openfeign`

## Springboot 项目风格

- 工具类的使用，优先使用项目内现有的工具类，其次使用`springboot`提供/默认集成的依赖（例如`StringUtils`,`BeanUtils`,`ObjectMapper`）
- 项目中引入了`mybatis/mybatis-plus/mybatis-flex`体系框架的情况下，优先使用`Mybatis`体系框架，而不是`JdbcTemplate`
    - 如果使用的是`mybatis-plus/mybatis-flex`等增强`mybatis`的框架，严禁滥用提供的构造器构造复杂SQL，而是应该是在XML文件中编写
    - `mybatis`严禁使用`@Select`/`@Update`/`@Delete`等注解在接口方法上编写复杂的语句，而是应该在XML文件中编写
- 依赖库，优先使用项目类引入的依赖，其次使用`springboot`提供/默认集成的依赖
    - 例如默认集成的`jackson`用来进行`JSON`处理,`RestTemplate`来进行外部请求，`JdbcTemplate`来执行SQL等
    - 当前缺少工具使用时，优先考虑`spring`体系是否已经包含了这些功能，直接使用，而不是额外引入其他依赖
- springboot-web项目默认都是REST接口，没有视图页面参与，纯粹的API接口，除非有特殊要求
- 默认国际化特征都是中国，东八区时区，除非有特殊需求
- AOP事务，当需要调用的目标方法上有事务注解时，应该使用对应的代理对象进行调用，而不是直接调用，避免直接使用出现事务失效
  - 事务注解只能添加在 `public` 公开方法上，严禁加到 `private`/`protected` 等方法上

## 代码安全

- 不要硬编码一一切的敏感信息，应该从配置文件或者环境变量中获取或者要求用户输入，例如：密码，`ApiKey`,`SecretKey`等信息
- 不要使用不安全的或者已经过时的加密/摘要算法，例如`MD5`,`SHA-1`这种不安全的算法
- 不要相信函数入参的可靠性/安全性，对函数入参需要进行严格的参数校验，避免不合法的输入（超出边界、不在取值范围内、异常的注入）
- 对于反序列化工具使用时，应该严格控制反序列化的行为，避免出现恶意反序列化执行导致恶意代码执行，例如XML反序列化，JSON反序列化等
- 对于脚本执行是，应该严格检查脚本内容的安全性，避免恶意脚本执行（删库，大批量文件删除，提取敏感信息，访问敏感文件等）
- 对于外部接口传入的路径请求（WEB输入参数，界面输入参数等），应该检查传入路径的安全性
    - 避免路径穿透（通过..绕过），敏感文件访问，转义逃逸(通过%00等转义序列)等安全风险
- 严格控制敏感文件访问，避免访问敏感文件（例如.ssh,passwd）等敏感文件
- SQL语句的执行，严格执行以预编译占位符方式执行语句，
    - 杜绝拼接SQL（含有不可信来源参数）造成的SQL注入风险，
    - 杜绝高危/敏感语句执行，例如`grant`,`drop`,`alter`等语句
    - 检查更新类(`delete`/`update`)语句必须包含有效的`where`条件，防止意外删除/更新

## 代码性能

- 合理运用缓存，Map，线程池，连接池，对象池等优化技术，在保证逻辑正确的前提下，尽可能提高运行性能
- 正确资源管理，对于大对象，流对象，连接对象，使用完之后应该立即释放资源，正确关闭或者释放
- 合理内存管理，不要无脑使用内存，特别是在使用内存当做缓存时，缓存大小应该具有限制或者定期失效策略，防止缓存无限制增大导致OOM或者频繁GC
- 如果项目使用了`spring-redis`或者`redisson-starter`时，可以优先考虑使用`redis`作为缓存，而不是内存作为缓存
    - 这主要是考虑多实例部署情况下的内存缓存与负载均衡的协调问题

## 代码命名与注释

- 代码命名需要具有可读性，需要能够根据命名就能大概知道其作用，运行逻辑，设计模式等信息，严禁含糊的命名或者无意义的命名
    - 例如getUserById就能知道获取是根据id获取user对象
    - 而不是getUser这种宽泛的命名
    - 当命名不足以表示含义或者具有二义性时，需要额外添加注释说明用途，特别是在成员变量上
- 代码注释，在总体逻辑不复杂不涉及复杂算法或者设计的时候，不需要注释或者只需要简短的注释用于描述主要步骤即可
    - 但是如果逻辑复杂或者涉及复杂算法或者设计的时候，应该简要的使用一段注释说明总体逻辑，之后每一关键步骤说明步骤核心逻辑
- for-i 型循环，如果循环变量在作用于内不会具有多个循环变量的情况下，直接使用`i,j,k`表示即可
    - 但是，如果循环嵌套，且循环变量具有特殊含义，就应该是一个具有可读性的命名，而不是`i,j,k`这种无意义的
    - 例如，对数组遍历，这时候循环变量仅仅是作为索引使用，没有特殊含义，因此使用`i,j,k`这种即可
    - 但是，如果遍历的时候，还需要在循环变量上进行运算的情况，就需要考虑具有可读含义的命名，例如`offset`等
- foreach 型循环，同样，如果在作用于内指代循环变量清晰的情况下，使用 `item` 作为变量名即可
    - 但是，如果多个foreach循环嵌套的情况或者可能造成混淆得情况下，就应该使用具有明确含义的变量名
- 对于controller中，仅使用一个`Pojo`对象接受前端的参数时，变量名尽量简短，因为这个变量名往往是没有什么特殊含义的
    - 一般叫做 `webVo` 即可

## 代码测试

- 需要编写单元测试代码，默认使用`JUnit4`进行单元测试，简单场景允许直接使用 main 函数方式测试
- 进行参数边界检查，范围检查，等效类检查，条件组合覆盖检查，检查是否正确实现逻辑
- 安全检查，恶意输入检查，检查是否能够防御恶意输入

## 文档编写

- 函数文档编写，函数文档编写时，需要说明函数作用
    - 每个入参类型以及取值范围并给出正确与错误的参数示例
    - 如果参数的值是从某个类中定义的常量选取，需要说明取值来源与范围
    - 说明正确与错误运行时的返回值
    - 可能抛出的异常
    - 如果输入某些值必定导致某些输出或者异常的情况下，需要特别说明
    - 最后，需要例举2-3个使用示例
- 接口文档编写，针对MVC的API接口，除了函数文档部分的要求之外
    - 额外需要添加请求路径的信息以及请求方式限定约定描述
    - 最后，需要给出使用`axios`作为前端请求库时的使用方式

## 异常抛出

- 一般在spring体系下的web项目中，都会使用统一的异常处理方式
    - 方法一为：`@RestControllerAdvice` 配合 `@ExceptionHandler` 来进行全局异常处理
    - 方法二为：继承 `WebExceptionHandler` 接口实现全局异常处理
- 这两种方法一般都会将异常进行统一的处理，一般都会分类处理异常
- 因此，需要先查看这两种实现都处理了哪些异常，选取合适的异常类型进行抛出
- 一般会有一个异常，专门用于将报错信息提供给用户查看，这个异常类型就是业务异常类型
    - 一般在其他的 `*ServiceImpl` 类中也会使用到，参考查看其他的用法进行使用

## Spring-Mvc API 编写

- 先查看项目内其他controller，查看都是用了哪些HTTP方法
    - 一般有两种，一种是 `GET`/`POST`/`PUT`/`DELETE` 四种构成的 `RESTful` 风格的 API
    - 一种是只使用 `GET`/`POST` 两种构成的封闭式风格 API
    - 确定现有的 API 风格之后，按照现有的风格编写API接口
        - 不要出现现有的风格是 GET/POST 两种的封闭式，而编写的接口却是 RESTful 风格的API
- 还要看其他 controller 是否继承了基类，如果其他的都继承了，那么新编写的也需要继承
- 接着看其他 controller 的返回值都是怎么什么类型表示的，是统一使用某个类型封装响应
    - 还是各种类型返回都行的
    - 如果都是同一个类型封装的，那么也要按照这样的返回值类型进行编写
- 分页的处理，先查看其他 controller 中针对分页是怎么处理的，分页参数的来源
    - 然后按照其他 controller 的处理方式处理分页请求
- controller 中只允许进行参数校验和调用service方法，不允许直接使用 dao/mapper 层的方法操作数据库
